## 📝 **ПРАВИЛА НАПИСАНИЯ ДОКУМЕНТАЦИИ**

### 1. **Именование файлов:**
- Используйте kebab-case: `order-management.md`
- Описательные имена: `telegram-bot.md`, не `bot.md`
- Группируйте по темам в подпапках

### 2. **Структура документа:**
```markdown
# Название сервиса/функции

## 🎯 Основные функции
- Список ключевых возможностей

## 🏗️ Архитектура
- Описание структуры

## 📊 База данных
- Схемы и таблицы

## 🔧 API Endpoints
- Список эндпоинтов

## 🎨 Интерфейс
- Описание UI компонентов

## 🚧 Статус разработки
- Что готово, что в процессе

## 📝 TODO
- Планы на будущее
```

### 3. **Использование эмодзи:**
- 🎯 - Цели и функции
- 🏗️ - Архитектура
- 📊 - Данные и база данных
- 🔧 - API и техническая часть
- 🎨 - UI/UX
- 🚧 - Статус разработки
- 📝 - Планы и TODO
- ⚠️ - Важные замечания
- ✅ - Готовые функции
- ❌ - Проблемы или ограничения

## 🔄 **ПРОЦЕСС ОБНОВЛЕНИЯ ДОКУМЕНТАЦИИ**

1. **При добавлении новой функции:**
   - Обновите соответствующий `.md` файл в `docs/`
   - Добавьте описание в `docs/README.md`

2. **При изменении API:**
   - Обновите `docs/api/endpoints.md`
   - Добавьте примеры в `docs/api/examples.md`

3. **При деплое:**
   - Обновите `docs/deployment/`
   - Добавьте запись в `docs/changelog/`
